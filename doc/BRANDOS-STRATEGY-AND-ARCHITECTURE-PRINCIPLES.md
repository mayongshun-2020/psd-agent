# BrandOS 产品战略与架构设计原则 V1.0

## 1. 项目定位

BrandOS 不是一个单纯的 AI 出图工具。

BrandOS 的核心定位是：

> AI 驱动的品牌设计操作系统（Brand Design Operating System）

其目标不是直接生成若干视觉结果，而是通过持续学习品牌资产，构建品牌知识库、设计规则引擎与设计生产工作流，最终实现品牌内容的自动化生产。

支持的内容类型包括：

- 商品详情页
- 品牌首页
- 活动专题页
- Banner
- A+ 页面
- 营销素材
- Figma 设计稿

最终目标是：

让 AI 理解品牌，而不仅仅是生成图片。

## 2. 核心价值

### 2.1 错误方向

如果产品路径只是：

商品图 -> AI 生成图片 -> 完成

那么本质上仍然停留在低壁垒的 AI 出图赛道，难以建立长期竞争力。

### 2.2 正确方向

BrandOS 应该围绕以下链路建立能力：

品牌资产 -> 品牌知识库 -> 设计规则引擎 -> 设计生产工作流 -> 自动生成品牌内容

因此，BrandOS 的核心价值不在于单次生成质量，而在于：

- 品牌一致性
- 设计效率提升
- 设计资产沉淀
- 品牌知识传承

## 3. 品牌规则体系设计

### 3.1 品牌规则漂移问题

持续上传官网、详情页、活动页、PSD 等资产后，如果系统直接重建规则或自动覆盖现有规则，会带来以下风险：

- 品牌规则污染
- 设计风格漂移
- 输出结果越来越混乱

因此，BrandOS 必须建立分层规则体系，而不能让所有新资产直接等价影响品牌规则。

### 3.2 三层规则模型

#### Core Rule

Core Rule 是品牌核心规则，也可以理解为品牌宪法。

主要包括：

- 品牌名称
- 品牌定位
- 主色
- 辅助色
- 字体规范
- Logo 规范
- 品牌调性
- 品牌设计语言

设计原则：

- 不可自动修改
- 必须人工审批
- 是所有设计生成的最高优先级约束

#### Derived Rule

Derived Rule 是面向不同页面类型或业务场景的页面规则。

例如：

- 商品详情页规则
- 品牌首页规则
- 618 活动页规则
- 双 11 活动页规则

设计原则：

- 可以随业务和页面类型调整
- 受 Core Rule 约束
- 用于承接页面层级的设计变化

#### Asset Memory

Asset Memory 是资产记忆库，用于保存历史品牌资产，例如：

- 官网
- PSD
- Banner
- 活动页
- 详情页

设计原则：

- 仅作为参考案例
- 不得直接修改品牌规则
- 主要用于检索、参考与辅助生成

### 3.3 权重设计

推荐采用如下权重：

- Core Rule：70%
- Derived Rule：20%
- Asset Memory：10%

这样可以在保持品牌稳定性的同时，允许业务层面进行有限调整。

## 4. 品牌规则版本管理

BrandOS 不应允许如下方式：

上传资产 -> 自动覆盖 `design.md`

必须采用版本控制机制，例如：

- Brand Rule V1.0
- Brand Rule V1.1
- Brand Rule V1.2

并支持：

- Diff 比较
- 审批
- 回滚
- 审计日志

### 4.1 规则变更流程

推荐流程为：

上传资产 -> AI 发现规则差异 -> 生成规则变更建议 -> 人工审批 -> 更新规则版本

### 4.2 设计原则

- 任何规则更新都必须可追溯
- 任何版本发布都必须可回滚
- 任何差异都必须可解释

## 5. 设计生成原则

### 5.1 页面不是图片

错误认知：

详情页 = 一张长图

正确认知：

详情页应由以下结构组成：

- 页面结构
- 组件布局
- 文案
- 视觉资源
- 设计规则

因此，所有页面都必须先生成中间结构，而不能直接生成最终图片。

例如：

```json
{
  "section": "hero",
  "layout": "left-text-right-image",
  "title": "AI智能识别污渍"
}
```

这类结构化结果，才是后续布局引擎、设计稿生成和编辑能力的基础。

### 5.2 Agent 不得自由发挥

BrandOS 不应采用完全开放式页面规划。

因为如果没有模板约束，会出现：

- 今天 8 个模块
- 明天 12 个模块
- 后天 5 个模块

这对企业设计管理是不可接受的。

因此，必须采用标准页面模板，例如：

- Hero
- Feature
- Technology
- Scenario
- Parameter
- Brand Story
- CTA

Agent 的职责是在模板范围内生成内容，而不是无限制地创造页面结构。

## 6. 设计稿生成策略

推荐优先级如下：

Figma -> HTML -> PNG -> PSD

原因在于：

- 企业设计协作正在逐步迁移至 Figma
- Figma 更适合协作、审阅和持续编辑
- HTML 有利于结构化表达和工程化承接
- PNG 仅适合作为最终预览结果
- PSD 主要作为兼容方案保留

因此，BrandOS 在设计稿生成策略上应明确：

- Figma 是优先方向
- PSD 是兼容方案

## 7. 设计反馈机制

第一阶段建议只记录设计师修改行为，例如：

- 标题变小
- 颜色调整
- 布局调整

当前阶段应明确禁止：

- 自动强化学习
- 自动修改规则
- 自动覆盖设计规范

原因在于，AI 无法准确理解设计师修改背后的业务上下文、审美偏好和投放目标。

因此，更合理的路径是：

先积累高质量反馈数据，后续再逐步构建设计反馈学习系统。

## 8. 最重要的数据资产

很多团队会误以为设计稿本身最值钱，但 BrandOS 长期最核心的数据资产并不是单张设计稿，而是：

- 品牌规则
- 设计反馈
- 设计决策过程
- 设计历史版本

例如：

为什么使用蓝色？

来源可能是：

品牌规范 V3

为什么标题使用 48px？

来源可能是：

历史高表现详情页统计结果

这些可解释的设计知识，将成为未来品牌设计大模型训练的核心资产。

## 9. 目标客户价值

### 9.1 品牌方

BrandOS 为品牌方带来的价值包括：

- 品牌一致性
- 设计知识沉淀
- 设计效率提升
- 新人快速上手

### 9.2 TP 代运营公司

BrandOS 为 TP 代运营公司带来的价值包括：

- 多品牌规则管理
- 自动生成详情页
- 自动生成活动页
- 设计师效率提升

因此，对外价值表达更适合强调：

提升设计生产效率 80%

而不是：

AI 帮你画图

## 10. 产品发展路线

### 10.1 阶段一

- 品牌知识库
- 详情页生成
- Figma 生成

### 10.2 阶段二

- 品牌首页生成
- 活动页生成
- 营销素材生成

### 10.3 阶段三

- 品牌设计工作流
- 审批体系
- 团队协作

### 10.4 阶段四

- 设计反馈学习
- 品牌专属模型
- 品牌数字设计师

## 11. 战略优先级

开发优先级建议如下：

1. 品牌知识库
2. 规则引擎
3. 布局引擎
4. Figma 生成
5. 图片生成

最终的核心原则始终是：

不要把 BrandOS 做成 AI 出图工具，而要把 BrandOS 做成一个让 AI 理解品牌的品牌设计操作系统。
